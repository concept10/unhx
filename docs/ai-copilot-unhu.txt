# UNHU — AI Copilot Prompting Guide

### *How to direct your AI assistant to begin building the UNHU kernel*

-----

> **What this guide is:** A structured set of prompts, context blocks, and strategies for using an AI coding assistant (GitHub Copilot, Claude, Cursor, Codeium, etc.) to actually begin implementing the UNHU microkernel project. Each section gives you the exact prompt language to use, what to expect back, and how to chain prompts into productive sessions.

-----

## 0. Before You Start — Priming the Copilot

AI assistants work best when they have full project context before you ask them to write code. Always begin a new session with the **Project Context Block** below. Paste it as your first message.

-----

### 📋 Project Context Block

*Copy and paste this at the start of every new AI session working on UNHU.*

```
You are working on UNHU (U Is Not Hurd, it's µ), an open-source operating 
system project. The goal is to build a true running microkernel based on 
the original Mach kernel design from Carnegie Mellon University (CMU, 1985–1990) 
and the Open Software Foundation (OSF MK series).

Core design rules you must follow in all code you write for this project:

1. KERNEL MINIMALITY — The kernel implements only: IPC (Mach ports), 
   virtual memory, tasks, threads, and scheduling. Nothing else belongs 
   in the kernel. BSD, filesystems, drivers, and networking all live in 
   userspace servers.

2. MACH IPC IS THE API — All inter-component communication goes through 
   Mach message-passing on ports. No shared kernel state between components.

3. PORTS AS CAPABILITIES — A port right IS authority. We do not add 
   separate permission checks on top of port rights. Holding a send right 
   to a port means you have permission to use that service.

4. DOCUMENT EVERY DECISION — All non-obvious design choices get a comment 
   citing either the CMU Mach 3.0 paper, the OSF MK documentation, or an 
   explicit note explaining why we deviate from historical Mach.

5. NO PREMATURE OPTIMIZATION — Do not collapse the kernel/server boundary 
   for performance without a measured benchmark proving it is necessary. 
   XNU collapsed BSD into the kernel; we will not repeat that.

Target: x86-64 bare metal and QEMU/KVM first. AArch64 second.
Language: C (kernel), Objective-C (framework layer above kernel).
Build system: CMake + Nix for reproducibility.

Current phase: Phase 1 — Kernel Core.
Immediate goal: A kernel that boots under QEMU, creates two tasks, 
and passes a Mach message between them via port IPC.
```

## 1. Repository Scaffolding

Use these prompts to get the AI to set up the directory structure and build system before writing kernel code.

-----

### Prompt 1.1 — Generate the repo skeleton

```
Using the UNHU project context above, create the initial directory 
structure for the kernel/ subdirectory. Include:

- kernel/ipc/      — Mach port and message implementation
- kernel/vm/       — virtual memory subsystem  
- kernel/kern/     — task, thread, scheduler
- kernel/platform/ — x86-64 hardware abstraction
- kernel/include/  — shared headers (mach/mach_types.h etc.)

For each directory, create a placeholder .c file and matching .h file 
with correct include guards and a comment explaining what that module 
will implement. Base the module descriptions on CMU Mach 3.0 design.

Also create:
- kernel/CMakeLists.txt configured for a freestanding x86-64 target
- kernel/linker.ld — a minimal linker script for a higher-half kernel
- A top-level README.md for the kernel/ directory explaining the 
  Mach design philosophy we are following
```

-----

### Prompt 1.2 — QEMU boot script

```
Create a shell script at tools/run-qemu.sh that:
1. Builds the UNHU kernel using CMake (out-of-source build in build/)
2. Launches QEMU with:
   - x86-64 machine type
   - 256MB RAM
   - Serial output to stdio
   - KVM acceleration if available, TCG fallback
   - A debugger stub on port 1234 (for GDB remote attach)
3. Include a second script tools/debug-qemu.sh that launches GDB 
   and connects to the QEMU stub, loading kernel symbols automatically.

Add comments explaining each QEMU flag used.
```

-----

### Prompt 1.3 — Initial boot entry

```
Write the x86-64 boot entry code for UNHU at kernel/platform/boot.S.

Requirements:
- Multiboot2 compliant header so GRUB can load us
- Set up a minimal GDT (null, code, data segments)
- Set up an initial stack of 16KB
- Clear the BSS section
- Jump to kernel_main() in C

Also write kernel/platform/gdt.c and kernel/platform/gdt.h with:
- GDT descriptor structures
- A function to load the GDT with lgdt
- Proper segment selector constants

Add a comment at the top of boot.S citing that this follows the 
Multiboot2 spec and referencing where kernel_main is defined.
```

## 2. Mach Core Types

Before implementing IPC or VM, the AI needs to generate the foundational type definitions that mirror CMU Mach 3.0.

-----

### Prompt 2.1 — Core Mach types header

```
Create kernel/include/mach/mach_types.h for UNHU.

This must define the core Mach 3.0 types exactly as the CMU design 
specified them. Include with proper documentation comments:

- mach_port_t          — a port name (integer index into a space)
- mach_port_right_t    — type of right (SEND, RECEIVE, SEND_ONCE, PORT_SET, DEAD_NAME)
- mach_msg_header_t    — message header structure
- mach_msg_body_t      — message body descriptor
- mach_msg_return_t    — message operation return codes (MACH_MSG_SUCCESS etc.)
- task_t, thread_t     — opaque handles
- vm_address_t, vm_size_t, vm_offset_t — VM address types
- kern_return_t        — general kernel return type with KERN_SUCCESS, 
                         KERN_INVALID_ARGUMENT, KERN_NO_SPACE etc.

Each type must have a comment explaining its role in the Mach design.
Reference the CMU Mach 3.0 paper (Accetta et al., 1986) in the file header.
```

-----

### Prompt 2.2 — Port space and port structures

```
Based on the mach_types.h we just created, implement the internal 
kernel structures for port management in:

- kernel/ipc/ipc_port.h  — struct ipc_port (the kernel-side port object)
- kernel/ipc/ipc_space.h — struct ipc_space (a task's port name space)
- kernel/ipc/ipc_entry.h — struct ipc_entry (one slot in a port space)

For ipc_port, include fields for:
- port type (active, dead, kernel)
- the message queue (a simple linked list for now)
- the receiver (the task holding the receive right)
- send right reference count
- a lock (spinlock for now)

For ipc_space, include:
- an array/table of ipc_entry slots
- the task that owns this space
- a lock

Add a comment on ipc_space explaining the distinction between port 
names (what userspace sees) and port pointers (what the kernel uses) 
— this is fundamental to Mach's security model.
```

## 3. IPC — The Heart of Mach

This is the most important part of the kernel. Use these prompts in sequence.

-----

### Prompt 3.1 — Message queue implementation

```
Implement a basic Mach message queue for UNHU in kernel/ipc/ipc_mqueue.c 
and kernel/ipc/ipc_mqueue.h.

The message queue sits inside each ipc_port and holds incoming messages 
for the port's receiver to consume.

Implement:
- struct ipc_mqueue — the queue structure with a linked list of messages,
  a count, a max depth limit, and a lock
- ipc_mqueue_init()      — initialize an empty queue
- ipc_mqueue_send()      — enqueue a message (copy semantics for now,
                           no in-place yet), return MACH_MSG_SUCCESS or 
                           MACH_SEND_NO_BUFFER if full
- ipc_mqueue_receive()   — dequeue a message into a caller buffer,
                           return MACH_MSG_SUCCESS or MACH_RCV_TOO_LARGE

Keep it simple: no blocking/sleeping yet. That comes in the thread 
scheduler phase. For now, send returns an error if the queue is full 
and receive returns an error if the queue is empty.

Explain in comments why real Mach uses a more complex scheme 
(blocking with thread wakeup) and mark it as a TODO.
```

-----

### Prompt 3.2 — mach_msg() system call entry

```
Implement the kernel entry point for the mach_msg() system call in 
kernel/ipc/ipc_kmsg.c.

This is the single most important system call in Mach. All IPC 
goes through it.

Implement a kernel-internal function:
  kern_return_t mach_msg_send(task_t sender, mach_msg_header_t *msg, 
                               mach_msg_size_t size);

It should:
1. Validate msg is a readable user pointer (basic bounds check for now)
2. Copy the message from user space into a kernel mach_msg buffer
3. Look up the destination port in the sender's ipc_space by port name
4. Verify the sender holds a SEND right to that port
5. Enqueue the message on the port's mqueue via ipc_mqueue_send()
6. Return appropriate kern_return_t

Add a comment block explaining the security invariant: a task can only 
send to ports for which it holds a send right. This is the capability 
model — port names are unforgeable within a task's space.

Mark clearly with TODO where we will add: out-of-line memory descriptors,
port rights in messages, and blocking semantics.
```

-----

### Prompt 3.3 — Kernel port (first real IPC test)

```
Create kernel/kern/kernel_task.c implementing a minimal kernel bootstrap.

We want to prove IPC works by having the kernel itself create two tasks 
and send a message between them at boot time as a smoke test.

Implement:
- kernel_task_init()  — creates the kernel task (task 0) with its own 
                        ipc_space
- create_test_tasks() — creates task_a and task_b, allocates a port in 
                        task_a's space with a receive right, passes a 
                        send right to task_b, then has task_b send a 
                        simple "hello" message to task_a, and prints to 
                        serial console to confirm receipt

This becomes the Phase 1 milestone test. When this prints:
  [UNHU IPC] message received: hello
  [UNHU] Phase 1 complete. Mach IPC operational.
...we know the kernel core works.

Call create_test_tasks() from kernel_main() after basic init.
```

## 4. Virtual Memory Subsystem

-----

### Prompt 4.1 — Physical memory manager

```
Implement a simple physical page allocator for UNHU in:
- kernel/vm/vm_page.h
- kernel/vm/vm_page.c

Use a free list (linked list of free page frames). 

Implement:
- struct vm_page        — represents one physical page frame
- vm_page_init(base, size) — initialize the allocator from a memory 
                              range (we'll get this from Multiboot2 map)
- vm_page_alloc()       — return one free page, or NULL if exhausted
- vm_page_free(page)    — return a page to the free list

Also create kernel/vm/vm_map.h declaring the vm_map structure 
(a task's virtual address space) — just the struct definition and 
forward declarations for now. Add a comment explaining that in Mach, 
vm_map entries are ranges with backing memory objects (external pagers), 
and mark the external pager protocol as a future TODO.
```

-----

### Prompt 4.2 — Page table setup (x86-64)

```
Implement x86-64 paging setup for UNHU in kernel/platform/paging.c.

Requirements:
- Set up a 4-level page table (PML4 → PDPT → PD → PT)
- Identity-map the first 4MB for the kernel (temporary)
- Map the kernel to the higher half at 0xFFFFFFFF80000000
- Enable CR0.PG
- Keep the physical memory allocator integrated 
  (allocate page table pages via vm_page_alloc())

Add detailed comments on each level of the page table walk. 
A lot of people get confused by the x86-64 paging structure — 
make this the clearest possible reference implementation.

Include a function:
  void paging_map(uint64_t virt, uint64_t phys, uint64_t flags);
for mapping individual pages, and:
  void paging_init(struct multiboot_mmap *mmap, uint32_t mmap_len);
for initial setup from the Multiboot2 memory map.
```

## 5. Task and Thread Model

-----

### Prompt 5.1 — Task structure

```
Implement the Mach task abstraction for UNHU in:
- kernel/kern/task.h
- kernel/kern/task.c

A Mach task is a unit of resource ownership: it has a virtual address 
space (vm_map) and a port namespace (ipc_space). It is NOT a thread.

struct task must include:
- task_id (integer)
- vm_map pointer (its address space)
- ipc_space pointer (its port namespace)
- a linked list of its threads
- a lock
- kernel_task boolean flag
- task state (RUNNING, SUSPENDED, HALTED)

Implement:
- task_create(parent_task) → new task inheriting address space structure
- task_destroy(task)       → clean up all resources
- task_suspend(task)       → suspend all threads
- task_resume(task)        → resume all threads

Add a comment explaining the distinction between tasks and threads 
in Mach: threads execute, tasks own resources. A task with no threads 
is valid (it just does nothing). This is different from UNIX processes 
where the process and its single thread are conflated.
```

-----

### Prompt 5.2 — Thread structure and context switch

```
Implement the Mach thread abstraction for UNHU in:
- kernel/kern/thread.h
- kernel/kern/thread.c
- kernel/platform/context_switch.S (x86-64 assembly)

A thread is a unit of execution. It belongs to exactly one task.

struct thread must include:
- thread_id
- owning task pointer
- CPU state (struct cpu_state — all x86-64 registers)
- stack pointer and stack base
- thread state (RUNNING, WAITING, BLOCKED, HALTED)
- scheduler fields (priority, quantum remaining)
- next/prev for the run queue linked list

Implement:
- thread_create(task, entry_point, stack_size) → allocates stack, 
  sets up initial register state so the thread will begin executing 
  at entry_point when scheduled
- thread_destroy(thread)
- thread_switch(from_thread, to_thread) → calls context_switch.S

In context_switch.S implement the actual register save/restore using 
PUSHQ/POPQ for all callee-saved registers, and document each register 
being saved and why.
```

-----

### Prompt 5.3 — Round-robin scheduler

```
Implement a simple round-robin scheduler for UNHU in:
- kernel/kern/sched.c
- kernel/kern/sched.h

For Phase 1, we just need something that works. Keep it simple.

Implement:
- A single global run queue (linked list of runnable threads)
- sched_enqueue(thread)    — add thread to run queue
- sched_dequeue()          — take the next thread off the front
- sched_tick()             — called from timer interrupt, decrements 
                             current thread's quantum, calls 
                             sched_yield() if expired
- sched_yield()            — switch from current thread to next in queue
- sched_init()             — initialize the scheduler, set up the 
                             timer interrupt (x86-64 PIT or LAPIC)

Add a comment block at the top acknowledging this is not the Mach 
scheduler (which uses a multi-level feedback queue with thread 
priorities and decay). Mark it as a TODO for Phase 2, and reference 
the Mach 3 scheduler paper (Tevanian et al.) as the design to follow.
```

## 6. Bootstrap Server

The bootstrap server is the first userspace process and the service registry for the entire system.

-----

### Prompt 6.1 — Bootstrap server skeleton

```
Create the initial bootstrap server for UNHU in servers/bootstrap/.

The bootstrap server is the first task started by the kernel. Every 
other server registers with it and looks up other servers through it. 
It is the Mach equivalent of /sbin/init — but it works entirely through 
port IPC, not system calls.

Implement:
- servers/bootstrap/bootstrap.c  — main entry point
- servers/bootstrap/registry.c   — a simple name→port mapping table
- servers/bootstrap/bootstrap.h  — public interface

The registry should support:
- bootstrap_register(name, port)   — server registers its service port
- bootstrap_lookup(name) → port    — client gets a send right to a service
- bootstrap_checkin(name) → port   — server claims a pre-registered slot

Add a comment explaining why the bootstrap server exists: when a new 
task is created, it has no port rights except one — a send right to 
the bootstrap port. Everything else is obtained by negotiating through 
bootstrap. This is how Mach enforces that capability distribution 
starts from a known root.
```

## 7. Chaining Sessions — Multi-Turn Prompting Strategy

AI assistants have limited context windows. Use these strategies to maintain momentum across sessions.

-----

### Session Handoff Template

At the end of each productive session, ask the AI:

```
Summarize what we implemented in this session as a structured 
status update I can paste at the start of the next session. Include:
1. Files created or modified
2. Functions implemented and their signatures
3. What works / what is stubbed
4. The next concrete task to tackle
5. Any open design questions we discussed
```

Paste the result at the beginning of the next session after the Project Context Block.

-----

### When the AI Gets Confused About Mach

If the AI starts producing Linux-style kernel code, or confuses Mach concepts with POSIX/Linux concepts, reset with this prompt:

```
Stop. You are conflating Mach and Linux/POSIX concepts. 

In UNHU / Mach:
- There are NO file descriptors in the kernel. FDs are a BSD server concept.
- There are NO sockets in the kernel. Networking is a userspace server.
- There is NO VFS in the kernel. Filesystems are translator/server processes.
- System calls do NOT directly access kernel subsystems. They go through 
  port IPC to the appropriate server.
- The kernel has exactly four jobs: IPC, VM, tasks/threads, scheduling.

Please re-read the project context block and redo the last response 
with proper Mach architecture.
```

-----

### Asking for Historical Accuracy

When you want the AI to ground its implementation in historical Mach:

```
Before writing this code, tell me: how did CMU Mach 3.0 implement 
[topic]? What were the key design decisions and trade-offs? 
Then implement UNHU's version, noting where we follow the original 
design and where we deviate and why.
```

-----

### Asking for a Code Review Against Mach Principles

```
Review the code in [file] against the five UNHU design principles 
in the project context. Flag any places where:
1. Logic that belongs in a userspace server has crept into the kernel
2. A performance shortcut has violated the kernel/server boundary
3. Security depends on something other than port right ownership
4. Design decisions are undocumented
5. The Mach abstraction has been compromised for convenience

Suggest fixes for each violation.
```

### Milestone: Kernel Boots (v0.1)

```
Write a test procedure for verifying UNHU v0.1 "kernel boots" milestone.
The test should:
1. Build the kernel with CMake
2. Launch under QEMU via tools/run-qemu.sh
3. Capture serial output
4. Verify the output contains our boot banner and "kernel_main entered"
5. Verify QEMU exits cleanly (or stays running, as expected)

Also write a GitHub Actions workflow at .github/workflows/boot-test.yml 
that runs this test on every push to main.
```

-----

### Milestone: IPC Works (v0.2)

```
Write a kernel self-test for the IPC milestone that:
1. Creates two kernel-mode tasks (task_a, task_b)
2. Allocates a port in task_a with receive right
3. Gives task_b a send right to that port
4. Has task_b send: { magic: 0xDEADBEEF, message: "phase1_ok" }
5. Has task_a receive and verify the magic number and message
6. Prints PASS or FAIL with details to serial console
7. Stores the result so kernel_main can halt with a clear pass/fail exit code

This test should live in kernel/tests/ipc_test.c and be called 
from kernel_main() in debug builds (controlled by a CMake option 
UNHU_BOOT_TESTS=ON).
```

## Appendix A — Recommended Reading for Your AI (paste these summaries)

When the AI seems unfamiliar with Mach internals, paste one of these:

### Mach IPC Summary (paste when working on IPC)

```
Mach IPC primer for this session:

A PORT is a protected message queue. The kernel owns ports.
A PORT RIGHT is a capability — an unforgeable token granting access to a port.
  - RECEIVE right: exclusive, one per port, lets you dequeue messages
  - SEND right: copyable, lets you enqueue messages  
  - SEND_ONCE right: like SEND but consumed after one use

Port rights live in a task's PORT SPACE — a table mapping port names 
(small integers the task sees) to actual kernel port objects 
(pointers the kernel uses). Tasks never see kernel pointers.

Messages are typed. A message can carry:
  - Inline data (raw bytes)
  - Port rights (moving capabilities between tasks)
  - Out-of-line memory (mapping VM regions into the receiver's space)

mach_msg() is the ONLY system call needed for IPC. 
It handles send, receive, or both (combined send+receive for RPC pattern).
```

### Mach VM Summary (paste when working on VM)

```
Mach VM primer for this session:

A VM_MAP is a task's virtual address space — an ordered list of 
vm_map_entry ranges, each backed by a vm_object.

A VM_OBJECT represents a source of data for pages. It can be:
  - A physical memory range (anonymous memory)
  - A file (backed by an external pager)
  - A device (backed by a device pager)

External pagers are USERSPACE SERVERS that supply pages on demand. 
The kernel asks a pager for a page via a Mach message. The pager 
responds with the data. The kernel maps it. This is how the kernel 
never needs to know about filesystems — it just talks to external 
pagers via IPC.

vm_allocate(), vm_deallocate(), vm_map(), vm_protect() are the 
key VM operations. They manipulate vm_map entries, not page tables directly.
```
